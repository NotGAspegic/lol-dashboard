Celery App Factory + Worker Config
~1.5 hrsSetup
  • Create backend/worker/celery_app.py. Instantiate Celery('lol_dashboard') and configure it with broker_url = settings.redis_url and result_backend = settings.redis_url. Keep this file minimal — it's the entry point only.
  • Set these Celery config values explicitly: task_serializer='json', result_serializer='json', accept_content=['json'], task_track_started=True, task_acks_late=True. The last two are critical — they prevent tasks from disappearing silently on worker crash.
  • Configure task_routes to separate workloads into two queues: 'worker.tasks.ingest.*' → queue 'ingestion' and 'worker.tasks.refresh.*' → queue 'refresh'. This lets you scale ingestion workers independently later.
  • Add the Celery worker service to docker-compose.yml: same image as the API, command celery -A worker.celery_app worker -Q ingestion,refresh --loglevel=info --concurrency=2. Add depends_on: [redis, postgres].
  • Smoke test: create a throwaway task @app.task def ping(): return "pong", run the worker with docker compose up worker, then from a Python shell call ping.delay() and verify you see it execute in the worker logs.
✦ Day 1 Deliverable

Celery worker container starts cleanly and processes a ping task. You see Task worker.tasks.ping succeeded in the worker logs.

The ingest_match Task — Core Unit of Work
~2 hrsCode
  • Create backend/worker/tasks/ingest.py. Write the ingest_match task using @shared_task(bind=True). The bind=True gives you access to self, which you need for retry logic.
  • The task body: (1) check if match_id already in DB — return early if so, (2) call rate_limiter.acquire(region), (3) fetch match + timeline via RiotClient, (4) validate with DTOs, (5) write to DB via your Phase 1 upsert ops, (6) commit. Wrap steps 2–5 in a single try/except.
  • Handle the three Riot error cases explicitly: 404 (match doesn't exist — log a warning and return, do NOT retry), 429 (rate limited — retry after 30s), 5xx (Riot server error — retry after 60s, max 3 times).
  • Celery tasks must be synchronous functions — you cannot use async def with a standard Celery worker. Wrap your async DB calls with asyncio.get_event_loop().run_until_complete(), or better: use gevent pool. For now, convert your DB upsert ops to sync SQLAlchemy (not async) inside the task context.
  • Test it: manually call ingest_match.delay("NA1_some_match_id", "na1") from a Python shell. Watch the worker logs. Verify the match lands in Postgres.
⚠

Celery tasks are synchronous by default. Do not use async def here — it won't work as you expect. Keep the task sync and call any async helpers through asyncio.run() or restructure your DB ops to have sync versions.

✦ Day 2 Deliverable

ingest_match.delay("real_match_id", "na1") executes in the worker and stores the match in Postgres. Running it twice on the same match skips cleanly on the second call.

The ingest_summoner_matches Task — Batch Dispatch
~2 hrsCode
  • Write ingest_summoner_matches task. Its job is NOT to fetch match data — it's an orchestrator. It: (1) fetches the match ID list from Riot, (2) filters out match IDs already in the DB, (3) dispatches one ingest_match subtask per new match ID.
  • Use ingest_match.apply_async(args=[match_id, region], countdown=i*3) for each dispatch — the countdown=i*3 staggers the tasks 3 seconds apart. This is your second line of defense against rate limit bursts (the Redis limiter is the first).
  • Accept a since_match_id optional param — when provided, only fetch match IDs newer than this cursor. This is how incremental syncing works: you don't re-fetch 500 old matches every time, just the new ones since the last sync.
  • After dispatching all subtasks, update summoner.match_history_cursor in the DB with the newest match ID in the batch. This cursor is what the next sync will use.
  • Test: call ingest_summoner_matches.delay("puuid-here", "na1", count=10). Verify in the worker logs that 10 ingest_match sub-tasks get dispatched with staggered countdown values. Verify they all complete and hit the DB.
# The orchestrator pattern — dispatch, don't execute @shared_task(bind=True) def ingest_summoner_matches(self, puuid: str, region: str, count: int = 20, queue: int = 420): match_ids = riot_client.get_match_ids_sync(puuid, region, count, queue) existing = db.get_existing_match_ids(set(match_ids)) new_ids = [m for m in match_ids if m not in existing] for i, match_id in enumerate(new_ids): ingest_match.apply_async( args=[match_id, region], countdown=i * 3, # stagger 3s apart queue='ingestion' ) if new_ids: db.update_match_cursor(puuid, new_ids[0]) # newest first return {'dispatched': len(new_ids), 'skipped': len(existing)}
✦ Day 3 Deliverable

One ingest_summoner_matches call fans out into 10 staggered ingest_match sub-tasks, all completing successfully. The summoner's cursor updates in the DB.

The refresh_summoner Task + Onboarding Hook
~1.5 hrsCode
  • Write refresh_summoner task. Its job: (1) call GET /lol/summoner/v4 to refresh the summoner's profile data (level, icon — these can change), (2) upsert the summoner row, (3) dispatch ingest_summoner_matches with since_match_id set to the stored cursor so only new games get ingested.
  • Write a separate onboard_summoner task for first-time lookups. This one has no cursor, so it fetches the last 100 matches (Riot's maximum per request) and dispatches all of them. This is only ever called once per summoner.
  • Wire the onboard_summoner task to your FastAPI endpoint: when POST /api/v1/summoners/search is called with a name that doesn't exist in the DB yet, call onboard_summoner.delay(game_name, tag_line, region) and immediately return a 202 Accepted response with a task ID. The client polls for completion.
  • Add a GET /api/v1/tasks/{task_id} endpoint that checks Celery's result backend for task status. Return {"status": "PENDING" | "STARTED" | "SUCCESS" | "FAILURE", "result": ...}. This is how the frontend will show a loading state during onboarding.

Celery Beat — Automatic Periodic Refresh
~1.5 hrsSetup
  • Add a beat service to docker-compose.yml: same image as the worker, command celery -A worker.celery_app beat --loglevel=info --scheduler django_celery_beat.schedulers:DatabaseScheduler. Actually, for simplicity: use the built-in beat_schedule config dict instead of the DB scheduler — no extra dependencies needed.
  • In celery_app.py, set app.conf.beat_schedule with one entry: 'refresh-all-summoners' runs every 6 hours via crontab(hour='*/6'). This task will query the DB for all tracked summoners and dispatch a refresh_summoner task for each.
  • Write the refresh_all_tracked_summoners periodic task. It queries SELECT puuid, region FROM summoners WHERE last_updated < NOW() - INTERVAL '6 hours', then dispatches refresh_summoner.apply_async(args=[puuid, region], queue='refresh') for each result. Stagger with countdown=i*10 to avoid a thundering herd.
  • Start the beat service: docker compose up beat. For testing, temporarily change the schedule to run every 2 minutes (use crontab(minute='*/2')). Verify in logs that refresh_all_tracked_summoners fires and dispatches jobs. Then revert to 6 hours.
# celery_app.py — beat schedule config from celery.schedules import crontab app.conf.beat_schedule = { 'refresh-all-summoners': { 'task': 'worker.tasks.refresh.refresh_all_tracked_summoners', 'schedule': crontab(hour='*/6', minute='0'), 'options': {'queue': 'refresh'}, }, } app.conf.timezone = 'UTC'
✦ Day 5 Deliverable

Beat service fires refresh_all_tracked_summoners on schedule. Worker picks up the dispatched refresh jobs and processes them. The pipeline now runs entirely without manual intervention.

Retry Logic + Dead Letter Handling
~2 hrsCode
  • Implement exponential backoff on ingest_match. Use self.retry(exc=exc, countdown=2 ** self.request.retries * 30). So retry 1 waits 30s, retry 2 waits 60s, retry 3 waits 120s. Set max_retries=3.
  • Handle Riot's 429 specifically: read the Retry-After header from the response and use that value as the countdown instead of your exponential value. Riot tells you exactly how long to wait — use that signal.
  • After max_retries is exceeded, Celery raises MaxRetriesExceededError. Catch this in a task on_failure callback and write the failed match ID + error reason to a failed_ingestions table in the DB. This is your dead letter queue — a permanent record of what failed and why.
  • Create the failed_ingestions table via a new Alembic migration: columns match_id, region, error_type, error_message, attempts, failed_at. Add an Alembic migration and run it.
  • Write a retry_failed_ingestions periodic task (runs daily at 3am via Beat) that reads from failed_ingestions and re-dispatches them. Cap at 7 days old — failures older than that are probably permanently gone from Riot's API.
✦ Day 6 Deliverable

Simulate a failure by temporarily breaking your Riot API key. Watch tasks retry with exponential backoff and ultimately land in failed_ingestions. Restore the key, run retry_failed_ingestions, watch them succeed and get removed from the failed table.

Bulk Insert Optimization — Timeline Frames
~2 hrsPerformance
  • Profile your current timeline insertion. Add timing logs around upsert_timeline_frames. A match with 30 minute frames × 10 participants = 300 rows. One-by-one inserts will take 300+ roundtrips — this is the biggest performance bottleneck in the whole pipeline.
  • Rewrite upsert_timeline_frames to use a single bulk insert: build a list of 300 dicts in Python, then call session.execute(insert(MatchTimelineFrame), rows_list). SQLAlchemy translates this to a single multi-row INSERT. This should be 20–50× faster.
  • Add ON CONFLICT DO NOTHING to the bulk insert so re-ingesting a match doesn't fail — it just silently skips existing rows. Use SQLAlchemy's insert(...).on_conflict_do_nothing(index_elements=['match_id', 'participant_id', 'frame_timestamp']).
  • Do the same optimization for upsert_participants — build a batch list and insert all 10 participants in one statement.
  • Re-run the timing logs. Target: full match ingestion (match + participants + 300 timeline frames) completes in under 200ms DB time.
# BEFORE — 300 individual inserts (slow) for frame in timeline_frames: session.add(MatchTimelineFrame(**frame)) # AFTER — 1 bulk insert (fast) from sqlalchemy.dialects.postgresql import insert stmt = insert(MatchTimelineFrame).values(timeline_frames) stmt = stmt.on_conflict_do_nothing( index_elements=['match_id', 'participant_id', 'frame_timestamp'] ) session.execute(stmt)
✦ Day 7 Deliverable

Ingesting a single match logs timeline insert: 18ms or similar — a dramatic improvement. Re-ingesting the same match is a no-op with no errors.

Response Caching with Redis
~1.5 hrsPerformance
  • Write a cache decorator in backend/utils/cache.py. It takes a ttl (seconds) and a key_prefix. On a cache hit it returns the Redis value deserialized from JSON. On a miss it calls the underlying function, serializes the result to JSON, stores it in Redis with SETEX, and returns the result.
  • Apply @cache(ttl=300, key="summoner:{puuid}") to the summoner profile endpoint. Apply @cache(ttl=300, key="matches:{puuid}:{page}") to the match list endpoint. These are the two most-called endpoints and benefit most from caching.
  • Implement cache invalidation: when a summoner's data is refreshed (new matches ingested), call redis.delete(f"summoner:{puuid}") and use redis.scan_match(f"matches:{puuid}:*") to wipe all their match list pages. Do this at the end of the refresh_summoner task.
  • Verify caching works: hit a summoner endpoint twice and log the timing. First call should hit the DB (50–100ms). Second call should return from Redis (<5ms). Log CACHE HIT and CACHE MISS at DEBUG level.
# utils/cache.py — simple Redis cache decorator import json, functools from config import redis_client def cache(ttl: int, key: str): def decorator(func): @functools.wraps(func) async def wrapper(*args, **kwargs): cache_key = key.format(**kwargs) cached = await redis_client.get(cache_key) if cached: return json.loads(cached) result = await func(*args, **kwargs) await redis_client.setex(cache_key, ttl, json.dumps(result)) return result return wrapper return decorator
✦ Day 10 Deliverable

Cached endpoints return in <5ms on the second hit. Cache invalidates correctly after a refresh — the third call goes back to the DB and re-populates the cache.
