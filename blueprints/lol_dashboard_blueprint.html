01Tech Stack Decisions & Rationale

Each choice is optimized for handling large-scale match data, serving real-time ML predictions, and rendering dense data visualizations without sacrificing DX.

Frontend
Next.js 14 + TypeScript

App Router for server-side data fetching per route. RSC reduces JS bundle for heavy data pages. TypeScript catches Riot API shape mismatches at compile time.

Styling
TailwindCSS + shadcn/ui

shadcn gives unstyled, accessible primitives you own. Tailwind allows rapid iteration on dense data layouts without CSS overhead.

Charts
Recharts + D3.js

Recharts for standard React charts (KDA line, win rate bar). D3 for custom visuals: gold curve brushes, hexbin matchup maps, force-graph synergy networks.

Backend API
FastAPI (Python 3.11+)

Async-native, automatic OpenAPI docs, Pydantic validation. 3× faster than Flask for concurrent Riot API calls. Native support for serving ML model predictions.

Task Queue
Celery + Redis Broker

Background workers for match ingestion respect Riot rate limits. Beat scheduler for periodic summoner refresh. Flower dashboard for queue monitoring.

Primary DB
PostgreSQL 16 + TimescaleDB

TimescaleDB hypertable on match_timeline_frames enables time-bucketing queries at millisecond speed across millions of frame rows.

Cache & Rate Limit
Redis 7

Token-bucket rate limiter for Riot API (20 req/s, 100 req/2min). Response cache for champion data (CDragon). Session store. Celery broker.

Machine Learning
scikit-learn + XGBoost + pandas

XGBoost for tabular classification (tilt, draft). scikit-learn pipelines for preprocessing. joblib for model serialization. No GPU needed at this scale.

Deployment
Docker Compose → Railway

Docker Compose for local dev parity. Railway for FastAPI + Celery workers + Redis. Vercel for Next.js frontend. GitHub Actions for CI/CD.

⚠ NOTE

Do NOT use TensorFlow/PyTorch for these models. XGBoost on tabular Riot data will outperform a neural network at this scale and be 10× easier to debug, explain, and deploy without a GPU server.

02System Architecture & Rate Limiting

The system has three distinct data flows: ingestion (background, rate-limited), query (user-triggered, cached), and prediction (on-demand, model-served).

Rate Limiting Strategy — Redis Token Bucket

Riot enforces two limits simultaneously: 20 requests/second and 100 requests/2 minutes. A dual-bucket Redis implementation handles both atomically.

# rate_limiter.py — Redis Lua script for atomic token bucket
import redis
import time

RATE_LIMIT_SCRIPT = """
local key_short = KEYS[1]
local key_long  = KEYS[2]
local now       = tonumber(ARGV[1])

-- Short window: 20 req/s
local short_count = redis.call('INCR', key_short)
if short_count == 1 then redis.call('EXPIRE', key_short, 1) end
if short_count > 20 then return {0, 'short'} end

-- Long window: 100 req/2min
local long_count = redis.call('INCR', key_long)
if long_count == 1 then redis.call('EXPIRE', key_long, 120) end
if long_count > 100 then return {0, 'long'} end

return {1, 'ok'}
"""

class RiotRateLimiter:
    def __init__(self, redis_client):
        self.r = redis_client
        self.script = self.r.register_script(RATE_LIMIT_SCRIPT)

    async def acquire(self, region: str) -> bool:
        result = self.script(
            keys=[f"rl:{region}:short", f"rl:{region}:long"],
            args=[int(time.time())]
        )
        if result[0] == 0:
            wait = 1 if result[1] == 'short' else 5
            await asyncio.sleep(wait)
            return await self.acquire(region)
        return True
ℹ INFO

Always check the X-Rate-Limit-Count and X-App-Rate-Limit headers in Riot API responses. Your local counter can drift from Riot's server-side counter due to network delays. Use the response headers as ground truth and throttle accordingly.

from celery import shared_task

@shared_task(bind=True, max_retries=3, default_retry_delay=60)
async def ingest_match(self, match_id: str, region: str):
    """Fetch, parse, and store a single match + its timeline."""
    try:
        await rate_limiter.acquire(region)
        match_dto   = await riot_client.get_match(match_id, region)
        timeline    = await riot_client.get_timeline(match_id, region)
        
        async with db_session() as session:
            await upsert_match(session, match_dto)
            await upsert_participants(session, match_dto)
            await upsert_timeline_frames(session, timeline)
            await session.commit()
            
    except RiotRateLimitError as e:
        raise self.retry(exc=e, countdown=30)

04Machine Learning Strategy

Model 1 — Tilt Detection
XGBClassifier

Concept: Compute a rolling window of features from a player's last N games and predict the probability they are currently "tilted" — defined as a mental state leading to continued performance degradation.

Ground Truth Labeling Strategy

This is the hardest part. You have two options:

Option A (Self-supervised): Label historical windows as tilt=1 when: (a) 3+ consecutive losses AND (b) KDA in the final 3 games is ≥20% below the player's personal baseline. This is objective and reproducible.

Option B (User-labeled): Add a "Was this a tilt session?" button to the UI. Collect ~200 labeled sessions per user, then fine-tune a personalized model. Better accuracy, requires user engagement.

Start with Option A for MVP. It generates labels from data you already have.

Feature Set (per 10-game rolling window)

consecutive_losses kda_slope death_count_trend gd_at_15_trend inter_game_interval_mean inter_game_interval_min vision_score_slope cs_per_min_slope game_duration_trend champion_variety hours_since_last_win total_games_today 
# feature_engineering/tilt.py
import numpy as np
from scipy.stats import linregress

def extract_tilt_features(recent_games: list[dict]) -> dict:
    """recent_games: last 10 match_participants rows, sorted desc by game_start"""
    
    kda_series     = [g['kills']+g['assists'] / max(g['deaths'],1) for g in recent_games]
    death_series   = [g['deaths'] for g in recent_games]
    gd15_series    = [g.get('gd_at_15', 0) for g in recent_games]
    starts         = [g['game_start_epoch'] for g in recent_games]
    
    x = np.arange(len(recent_games))
    
    # Linear regression slopes as trend features
    kda_slope,   *_ = linregress(x, kda_series)
    death_slope, *_ = linregress(x, death_series)
    gd15_slope,  *_ = linregress(x, gd15_series)
    
    # Time between games (minutes)
    intervals = [(starts[i]-starts[i+1])/60 for i in range(len(starts)-1)]
    
    consecutive_losses = 0
    for g in recent_games:
        if not g['won']: consecutive_losses += 1
        else: break
    
    return {
        'consecutive_losses':    consecutive_losses,
        'kda_slope':             kda_slope,
        'death_trend':           death_slope,
        'gd15_trend':            gd15_slope,
        'inter_game_mean_mins':  np.mean(intervals) if intervals else 0,
        'inter_game_min_mins':   np.min(intervals)  if intervals else 0,
        'champ_variety':         len(set(g['champion_id'] for g in recent_games)),
    }

Model 2 — Draft Win Probability
XGBClassifier

Concept: Given the 10 champion picks (5v5) and the target player's historical performance on their champion, predict the probability of winning the game before it starts.

Key insight: Encode team compositions as aggregated semantic scores (poke, engage, teamfight, peel, split), NOT raw one-hot champion vectors. 168+ champions makes one-hot sparse and patch-sensitive. Semantic scores are patch-resilient.

COMP_TAGS = {

def team_comp_scores(champion_ids: list[int]) -> dict:
    champions = [champ_name_from_id(c) for c in champion_ids]
    return {
        tag: sum(1 for c in champions if c in members)
        for tag, members in COMP_TAGS.items()
    }

Training Data Requirements

You need minimum 50,000 ranked matches for a reliable model. Collect via the Riot match list API across multiple summoners. Stratify by patch version to avoid temporal leakage (don't train on patch 14.12 and test on 14.1).

⚠ CRITICAL

Avoid target leakage. None of your draft features should contain information only available after the game starts (e.g., actual items built, real KDA). Features must be inferable purely from champion selections + historical player stats.

Model 3 — Power Spike Forecasting
KMeans + Regression

Concept: Use timeline frame data to learn the characteristic "gold curve shape" for a player on a specific champion. Then predict the minute window where a player is most likely to hold a meaningful gold advantage.

Approach

1. For each (player, champion) pair, collect all timeline curves as arrays of gold values at minutes 5, 10, 15, 20, 25, 30.

2. Normalize curves by dividing by game-end gold (shape matters, not magnitude).

3. Apply KMeans (k=3) to identify natural clusters: "early spiker", "late scaler", "consistent grinder".

4. Within each cluster, use linear regression to predict the peak gold-advantage minute given champion + items at 10 minutes.

5. Surface this as: "Based on your history, you tend to peak at minutes 18–24. Prioritize objectives in this window."
