Tilt Detection — Feature Engineering
~2 hrsData
  • Create backend/ml/features/tilt_features.py. This module takes a DataFrame of a player's recent matches (sorted newest first) and computes a rolling window of features from their last N games. The output is one feature row per "snapshot" — a point in time where you're assessing if the player is tilted.
  • Implement compute_tilt_features(games_df, window=10). For each game at index i, look at games [i, i+window) and compute: consecutive_losses (count from the start until first win), kda_slope (linear regression slope of KDA over the window), death_trend (slope of deaths), inter_game_minutes_mean (avg minutes between game starts), inter_game_minutes_min (shortest gap — rapid queuing indicator), champ_variety (unique champions played).
  • Compute KDA for each game: df['kda'] = (df['kills'] + df['assists']) / df['deaths'].clip(lower=1). Compute slopes using scipy.stats.linregress — fit a line to the KDA values in the window, the slope tells you if performance is improving or degrading over time.
  • Generate the tilt label using Option A (self-supervised): label a window as tilt=1 when BOTH: (a) consecutive_losses ≥ 3 AND (b) the window's avg KDA is ≥ 20% below that player's overall career KDA. This is objective and reproducible from data you already have.
  • Test the feature extraction: load your CSV, filter to one summoner's games, sort by timestamp, call compute_tilt_features. Print the first 5 feature rows. Verify consecutive_losses matches what you'd expect from looking at the raw wins/losses column.
# tilt_features.py — core feature computation from scipy.stats import linregress import numpy as np def slope(series): """Linear regression slope — positive = improving, negative = declining.""" x = np.arange(len(series)) if len(series) < 2: return 0.0 s, *_ = linregress(x, series) return float(s) def consecutive_losses(win_series): """Count losses from the start of the series until first win.""" count = 0 for w in win_series: if not w: count += 1 else: break return count def compute_tilt_features(games_df, window=10): games_df = games_df.sort_values('gameStartTimestamp', ascending=False).reset_index(drop=True) games_df['kda'] = (games_df['kills'] + games_df['assists']) / games_df['deaths'].clip(lower=1) rows = [] for i in range(len(games_df) - window): w = games_df.iloc[i:i+window] gaps = games_df['gameStartTimestamp'].iloc[i:i+window].diff().abs() / 60000 rows.append({ 'consecutive_losses': consecutive_losses(w['win'].tolist()), 'kda_slope': slope(w['kda'].values), 'death_trend': slope(w['deaths'].values), 'inter_game_mean': gaps.mean(), 'inter_game_min': gaps.min(), 'champ_variety': w['championId'].nunique(), 'win_rate_window': w['win'].mean(), }) return rows

Tilt Detection — Train + Evaluate Model
~2 hrsML
  • Create backend/ml/training/train_tilt.py. Load the CSV exports, run the feature extraction across all summoners in the dataset, combine into a single DataFrame with a tilt label column. Print the label distribution — expect 5–15% tilt=1 (imbalanced is normal).
  • Split the data by time — NOT randomly. Find the median gameStartTimestamp in the dataset, use everything before it as training, everything after as test. This prevents temporal data leakage where future games teach the model about past tilt patterns.
  • Build the sklearn pipeline: Pipeline([('scaler', StandardScaler()), ('clf', XGBClassifier(n_estimators=200, max_depth=4, learning_rate=0.05, scale_pos_weight=8, use_label_encoder=False, eval_metric='logloss'))]). The scale_pos_weight=8 compensates for class imbalance — tilt events are rare.
  • Evaluate with roc_auc_score on the held-out test set. Target AUC ≥ 0.68. Also print the confusion matrix and classification report. Don't chase high accuracy — the dataset is imbalanced so a model that always predicts "not tilted" gets 90%+ accuracy but 0.5 AUC. AUC is the correct metric here.
  • Serialize the trained pipeline: joblib.dump(pipeline, 'ml/models/tilt_v1.pkl'). Also save the feature names list as a JSON file — you'll need it when building the prediction endpoint to ensure features are passed in the correct order.
⚠

If your AUC comes back at exactly 0.5 — the model is no better than random. This usually means the tilt label is too rare (less than 3% of windows) or the labeling criteria is too strict. Try relaxing to consecutive_losses ≥ 2 and see if AUC improves.

✦ Day 3 Deliverable

Tilt model trained and saved as tilt_v1.pkl. AUC ≥ 0.65 on held-out test set. Feature names JSON saved alongside the model.

SHAP Explanations for Tilt Model
~1.5 hrsML
  • SHAP (SHapley Additive exPlanations) tells you WHY the model made a prediction — which features pushed the tilt score up or down. This is what makes the prediction feel trustworthy to a user. Without explanations, "You are 73% tilted" is useless. With SHAP: "Your KDA has dropped 3 games in a row and you're queuing faster than usual" is actionable.
  • In your prediction function, after getting the tilt probability, compute SHAP values: explainer = shap.TreeExplainer(pipeline.named_steps['clf']). Pass the scaled features: scaled_X = pipeline.named_steps['scaler'].transform([feature_vector]). Then shap_values = explainer.shap_values(scaled_X).
  • Convert SHAP values to human-readable reasons. Map each feature to a plain-English template: consecutive_losses → "You've lost {v} games in a row", kda_slope → "Your KDA has been {'improving' if v > 0 else 'declining'} recently", inter_game_min → "You've been queuing very quickly (min gap: {v:.0f} min)". Return the top 3 SHAP contributors as reason strings.
  • Create backend/ml/features/shap_reasons.py with the top_reasons(shap_vals, feature_names, feature_values, n=3) function. It sorts features by absolute SHAP value, picks the top N, and returns formatted reason strings.
  • Test it manually: take a summoner with 3+ consecutive losses, compute their feature vector, get predictions + SHAP reasons, print them. The reasons should accurately describe their recent game history.
# shap_reasons.py FEATURE_TEMPLATES = { 'consecutive_losses': lambda v: f"Lost {int(v)} games in a row", 'kda_slope': lambda v: f"KDA {'declining' if v < 0 else 'improving'} ({v:+.2f}/game)", 'death_trend': lambda v: f"Deaths {'rising' if v > 0 else 'falling'} recently", 'inter_game_min': lambda v: f"Queuing very fast (min gap: {v:.0f}m)", 'win_rate_window': lambda v: f"Recent win rate: {v*100:.0f}%", 'champ_variety': lambda v: f"Playing {int(v)} different champions", } def top_reasons(shap_vals, feature_names, feature_values, n=3): pairs = sorted( zip(feature_names, shap_vals[0], feature_values), key=lambda x: abs(x[1]), reverse=True ) reasons = [] for fname, shap_val, fval in pairs[:n]: template = FEATURE_TEMPLATES.get(fname) if template: reasons.append(template(fval)) return reasons

Draft Model — Feature Engineering
~2 hrsData
  • Create backend/ml/features/draft_features.py. The draft model predicts win probability from the 10 champion picks before a game starts. Key constraint: features must only use information available at draft time — no in-game stats, no items, no actual KDA.
  • Build a champion tag mapping from Data Dragon. Fetch https://ddragon.leagueoflegends.com/cdn/{patch}/data/en_US/champion.json and extract the tags field per champion (e.g. Yasuo = ["Fighter", "Assassin"]). Store as a JSON file in ml/data/champion_tags.json.
  • Define team composition scores. For each team of 5, count how many champions have each tag: ally_fighter_count, ally_mage_count, ally_support_count, ally_marksman_count, ally_tank_count, ally_assassin_count. Do the same for the enemy team. These 12 features encode team composition without one-hot encoding 168+ champions.
  • Add player-specific features: player_games_on_champ (how many ranked games on this champion), player_winrate_on_champ (win rate on this champion), player_kda_on_champ (avg KDA on this champion). These come from your match_participants history. For the training set, compute these at the time of the game — only use games before the current one.
  • Add patch version as a feature. Encode gameVersion as a float: "16.8.764" → 16.8. Champions are stronger/weaker on different patches — including this helps the model account for meta shifts.
ℹ

The composition tag approach is patch-resilient. One-hot encoding 168 champions means adding a new champion breaks your model. Tag-based scores degrade gracefully — a new champion just gets classified by its tags, same as any other.

✦ Day 6 Deliverable

champion_tags.json created. compute_draft_features(game_row, history_df) returns a consistent feature vector. Spot-check: Yasuo should have fighter and assassin scores, not support.

Draft Model — Train + Evaluate
~1.5 hrsML
  • Create backend/ml/training/train_draft.py. Load the draft CSV. Time-based split: train on first 70% of matches (sorted by timestamp), test on last 30%. This simulates deploying the model after collecting initial data and predicting on future games.
  • Train an XGBClassifier with: n_estimators=300, max_depth=3, learning_rate=0.05, subsample=0.8, colsample_bytree=0.8. The shallow trees (max_depth=3) prevent overfitting on a small dataset. The subsampling adds regularization.
  • Evaluate: target AUC ≥ 0.58 on the held-out test set. Anything above 0.55 is genuinely better than random (0.50) for draft prediction — game outcomes have huge inherent randomness that no model can capture. The ceiling for this type of model is around 0.65–0.70 even with perfect features.
  • Print feature importances: pd.Series(model.feature_importances_, index=feature_names).sort_values(ascending=False).head(10). The most important features should be player-specific (winrate on champ, games on champ) — this confirms the model learned that player familiarity matters more than raw champion composition.
  • Serialize: joblib.dump(model, 'ml/models/draft_v1.pkl'). Also save the exact feature column order as JSON — this is critical. The prediction endpoint must pass features in the exact same order the model was trained on.
