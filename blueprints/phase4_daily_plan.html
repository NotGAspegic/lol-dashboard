TimescaleDB Gold Curve Endpoint
~2 hrsBackend
  • Verify your match_timeline_frames hypertable has data. Run: SELECT COUNT(*), MIN(minute), MAX(minute) FROM match_timeline_frames; — you should see tens of thousands of rows spanning minutes 0–50+.
  • Add GET /api/v1/summoners/{puuid}/gold-curves to backend/api/v1.py. Accept optional query param champion_id. Returns average gold per minute for the player across all their games (filtered by champion if provided), and a position average for comparison.
  • The query: join match_timeline_frames + match_participants on match_id + participant_id. Filter by puuid. Group by minute. Aggregate: AVG(total_gold) per minute. Order by minute ASC. This gives you a smooth average gold curve across all games.
  • The tricky part: match_timeline_frames uses match_id (integer gameId) and participant_id (1–10), but match_participants has puuid. You need to join on match_id = gameId AND find which participant_id corresponds to the puuid in each match. Query match_participants first to get a (gameId, participant_id) mapping, then use it in the timeline query.
  • Return format: [{ minute: 0, avg_gold: 500 }, { minute: 1, avg_gold: 620 }, ...]. Cap at minute 35 — games beyond that are rare and skew the average. Filter out remake matches (gameDuration < 300).
  • Test with curl: curl "http://localhost:8000/api/v1/summoners/{puuid}/gold-curves". You should get ~35 data points. Verify minute 0 is ~500 gold and minute 10 is ~3000–4000 gold — those are realistic values.

Vision Score Impact Endpoint + Chart
~2 hrsBackend + Chart
  • Add GET /api/v1/summoners/{puuid}/vision-impact. Query: group all games by vision score quartile (Q1: 0–25th percentile, Q2: 25–50th, Q3: 50–75th, Q4: 75–100th). For each quartile, return: quartile, avg_vision_score, win_rate, game_count.
  • To calculate quartiles in PostgreSQL: use NTILE(4) OVER (ORDER BY visionScore) as a window function inside a subquery. The outer query groups by the NTILE bucket and aggregates win rate. This is pure SQL — no Python post-processing needed.
  • Create src/components/charts/VisionImpactChart.tsx. Use a Recharts BarChart with 4 bars — one per quartile. X-axis: quartile label ("Low Vision", "Below Avg", "Above Avg", "High Vision"). Y-axis: win rate %. Color bars by win rate using the same green/yellow/red thresholds as the champion table.
  • Add a data label inside each bar showing the average vision score for that quartile: e.g. "avg: 24". Use Recharts LabelList with position="insideTop". This immediately tells the user what vision score they need to be in each bucket.
  • Add this chart to the Timeline tab alongside the gold curve. Layout: gold curve on top (full width), vision impact chart below (full width). Both charts live in the same tab.
# vision impact query — NTILE window function vision_sql = text(""" WITH quartiled AS ( SELECT win, "visionScore", NTILE(4) OVER (ORDER BY "visionScore") AS quartile FROM match_participants WHERE puuid = :puuid ) SELECT quartile, ROUND(AVG("visionScore")::numeric, 1) AS avg_vision, ROUND(AVG(CASE WHEN win THEN 1.0 ELSE 0.0 END) * 100, 1) AS win_rate, COUNT(*) AS game_count FROM quartiled GROUP BY quartile ORDER BY quartile """) result = await session.execute(vision_sql, {"puuid": puuid})

Champion Matchup Data — Backend
~2 hrsBackend
  • Add GET /api/v1/summoners/{puuid}/matchups. This is the hardest query in Phase 4. Goal: for each game, find which champion was in the same position on the opposite team (the lane opponent). Return win rate against each enemy champion with ≥ 3 encounters.
  • Finding the lane opponent: for a given game and player position (e.g. JUNGLE), find participants where gameId = same game AND individualPosition = same position AND teamId != player's teamId. That participant's championId is the matchup champion.
  • Write this as a self-join on match_participants: alias the table twice — mp_player (filtered by puuid) and mp_enemy (same gameId, same position, different team). Join on mp_player.gameId = mp_enemy.gameId AND mp_player.individualPosition = mp_enemy.individualPosition AND mp_player.teamId != mp_enemy.teamId.
  • Aggregate: group by mp_enemy.championId. Return: enemy_champion_id, games, wins, win_rate, avg_kda_in_matchup. Filter to ≥ 3 games. Order by games DESC.
  • Test the endpoint. Verify results make sense — if you main Naafiri jungle you should see Vi, Lee Sin, Jarvan etc. as frequent opponents. Win rates should vary — you likely perform differently against different junglers.
# matchup self-join query mp_player = aliased(MatchParticipant, name="mp_player") mp_enemy = aliased(MatchParticipant, name="mp_enemy") result = await session.execute( select( mp_enemy.championId.label("enemy_champion_id"), func.count().label("games"), func.sum(case((mp_player.win, 1), else_=0)).label("wins"), (func.avg(case((mp_player.win, 1.0), else_=0.0)) * 100).label("win_rate"), ) .join(mp_enemy, (mp_player.gameId == mp_enemy.gameId) & (mp_player.individualPosition == mp_enemy.individualPosition) & (mp_player.teamId != mp_enemy.teamId)) .where(mp_player.puuid == puuid) .group_by(mp_enemy.championId) .having(func.count() >= 3) .order_by(func.count().desc()) )

Playstyle Fingerprint — Backend Endpoint
~1.5 hrsBackend
  • Add GET /api/v1/summoners/{puuid}/playstyle. Returns 6 normalized scores (0–100) representing the player's playstyle axes: Aggression, Farming, Vision, Objective Control, Teamfight, Consistency.
  • Calculate each axis from your existing data: Aggression = normalized avg kills/game. Farming = normalized avg cs_per_min (from challenges JSONB). Vision = normalized avg visionScore. Objective Control = normalized kill_participation average. Teamfight = normalized avg assists/game. Consistency = 100 - (stddev of KDA × 10) — lower variance = higher consistency.
  • Normalize each value to 0–100 using min-max normalization based on reasonable bounds. Example for kills: 0 kills/game = 0, 15+ kills/game = 100. For vision score: 0 = 0, 60+ = 100. These bounds are position-agnostic approximations — good enough for a visual fingerprint.
  • Return format: { aggression: 72, farming: 45, vision: 38, objective_control: 61, teamfight: 55, consistency: 68 }. Cache with TTL 600s (10 min — changes slowly).
  • Test: curl the endpoint for your main summoner. Values should roughly match your perception of your playstyle. A jungler should have higher objective control than a mid laner. A farm-heavy player should have high farming score.
✦ Day 9 Deliverable

Playstyle endpoint returns 6 scores between 0–100. Values feel qualitatively correct for the summoner's known playstyle.
