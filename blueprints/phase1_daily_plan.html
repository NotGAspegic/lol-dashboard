Docker Compose + PostgreSQL + Redis Up
~1.5 hrs Setup

Pydantic DTOs — Riot API Shapes
~2 hrs Code
  • Read the Riot API docs for these 3 endpoints: GET /lol/summoner/v4, GET /lol/match/v5/matches/{id}, GET /lol/match/v5/matches/{id}/timeline. Open the response in a browser first.
  • Create backend/models/riot_dtos.py. Write SummonerDTO with fields: puuid, id (encryptedSummonerId), profileIconId, summonerLevel.
  • Write ParticipantDTO — this is the big one. Include all fields from the blueprint schema: kills, deaths, assists, totalDamageDealtToChampions, goldEarned, visionScore, championId, teamId, individualPosition, win, challenges (as dict), perks (as dict).
  • Write MatchDTO wrapping MatchInfoDTO (gameId, gameDuration, gameStartTimestamp, queueId, gameVersion) + list of ParticipantDTO.
  • Write TimelineFrameDTO and ParticipantFrameDTO with: participantId, currentGold, totalGold, xp, level, minionsKilled, jungleMinionsKilled, position (x, y).
✦ Day 3 Deliverable

All DTOs importable. Paste a raw Riot API JSON response into a Python shell and validate it: MatchDTO.model_validate(raw_json) succeeds.

RiotClient — httpx Async Client
~2 hrs Code
  • Create backend/riot/client.py. Write a RiotClient class that holds an httpx.AsyncClient with X-Riot-Token header and a 10s timeout.
  • Implement regional routing: Riot uses two URL patterns — platform (na1.api.riotgames.com) for summoner/ranked endpoints, and regional (americas.api.riotgames.com) for match endpoints. Write a helper that maps a platform string like "na1" to the correct regional cluster.
  • Implement get_summoner_by_riot_id(game_name, tag_line, region) → calls /riot/account/v1/accounts/by-riot-id/{gameName}/{tagLine} on the regional cluster, then /lol/summoner/v4/summoners/by-puuid/{puuid} on the platform cluster.
  • Implement get_match_ids(puuid, region, count=20, queue=420) → returns a list of match ID strings. Queue 420 = Solo Ranked.
  • Implement get_match(match_id, region) and get_timeline(match_id, region). Both return raw dicts — validation happens in the caller, not here. Add error handling for 404 (match not found) and 429 (rate limited).
  • Quick test: write a __main__ block that fetches your own summoner data and prints it. Run it: python -m riot.client.
✦ Day 4 Deliverable

You can fetch your own match list from Riot's live API and print the first 5 match IDs to the console.

Redis Rate Limiter
~1.5 hrs Code
  • Create backend/riot/rate_limiter.py. Implement the dual-bucket Redis Lua script from the blueprint. Register it on the Redis client with register_script().
  • Wrap RiotClient's request methods so every call goes through await rate_limiter.acquire(region) before hitting the API. The limiter should sleep and retry automatically — callers should never need to think about rate limits.
  • Also read the X-App-Rate-Limit-Count header from each Riot response. If the server-side count is within 5 of the limit, sleep 1 second proactively. This prevents accidental 429s from clock skew.
  • Test it: write a loop that fires 25 requests in a row to get_match_ids and verify no 429 responses come back. Log the sleep events.
✦ Day 5 Deliverable

25 consecutive API calls complete without a 429. You see rate-limit sleep logs in the console when the short bucket fills.

SQLAlchemy Models + Alembic Init
~2 hrs Database
  • Create backend/models/db.py. Write SQLAlchemy ORM models for Summoner, Match, MatchParticipant. Use mapped_column with explicit types matching the schema blueprint exactly.
  • Write the MatchTimelineFrame model — this is the hypertable. Add a composite index on (match_id, participant_id, minute). Mark it explicitly so Alembic doesn't try to create a PK-based index instead.

  • Run alembic upgrade head. Then manually run the TimescaleDB hypertable conversion: SELECT create_hypertable('match_timeline_frames', 'frame_timestamp'); via psql or a migration script.
✦ Day 6 Deliverable

DB Utility Functions — Upsert Logic
~2 hrs Database
  • Create backend/db/ops.py. Write upsert_summoner(session, dto: SummonerDTO) using SQLAlchemy's insert().on_conflict_do_update() on puuid. Never use plain INSERT — matches can be re-fetched.
  • Write upsert_match(session, dto: MatchDTO). Compute winning_team from participants[x].win == True → teamId before inserting.
  • Write upsert_participants(session, match_id, participants: list[ParticipantDTO]). Compute derived fields here in Python before writing: cs_per_min = (minionsKilled + neutralMinionsKilled) / (gameDuration / 60), damage_share, kill_participation.
  • Write upsert_timeline_frames(session, match_id, timeline_dto). Iterate over all frames → all participant frames. Compute minute = frame_timestamp // 60000. Bulk insert with session.execute(insert(MatchTimelineFrame), rows) — do NOT use one insert per row.
  • Add a guard: before inserting any match, check if match_id already exists. Skip entirely if it does. This makes ingestion fully idempotent — safe to re-run anytime.
✦ Day 7 Deliverable

Manually call upsert_match() + upsert_participants() + upsert_timeline_frames() with a real Riot API response. Verify rows appear in all four tables via psql.

Wire It Together — Ingestion Pipeline Function
~1.5 hrs Code
  • Create backend/ingestion/pipeline.py. Write an async def ingest_match(match_id, region) function that: (1) checks if already stored, (2) fetches match + timeline via RiotClient, (3) validates with DTOs, (4) writes to DB via upsert ops, (5) commits session.
  • Write async def ingest_summoner(game_name, tag_line, region, count=20) that: fetches the summoner, upserts them, gets their last N match IDs, then calls ingest_match for each one sequentially.
  • Add proper logging throughout: use Python's logging module, not print(). Log at INFO level for each match stored, WARNING for skipped (already exists), ERROR for failed fetches.
  • Wrap the whole ingestion in a try/except. On httpx.HTTPStatusError with status 429, sleep 5 seconds and retry up to 3 times. On any other error, log and continue to the next match — don't crash the whole run.
✦ Day 8 Deliverable

ingest_summoner("YourName", "TAG", "na1", count=5) runs cleanly, stores 5 matches with all participants and timeline frames. No crashes, clean logs.
